Java设计模式：抽象工厂模式

简介

抽象工厂模式是一种创建型设计模式， 它能创建一系列相关的对象，而无需指定其具体类。

抽象工厂模式适合应用场景

  • 当代码需要与多个不同系列的相关产品进行交互时，但又不想基于具体产品类构建代码，可能是因为无法提前获取相关信息或考虑到未来的扩展性。在这种情况下，可以考虑使用抽象工厂模式。

  • 抽象工厂模式提供了一个接口，用于创建每个系列产品的对象。通过使用这个接口来创建对象，可以确保生成的产品与应用程序已有的产品类型保持一致，避免了与应用程序不兼容的产品类型的生成。

  • 另外，当一个类基于一组抽象方法而导致其主要功能变得不明确时，也可以考虑使用抽象工厂模式。在良好设计的程序中，每个类应该只关注一个单一的职责。如果一个类需要与多种类型的产品进行交互，可以将工厂方法抽取到独立的工厂类或具备完整功能的抽象工厂类中。

  • 使用抽象工厂模式的好处之一是，当向程序中添加新的产品或产品系列时，无需修改现有的代码。这提供了更好的扩展性和可维护性，同时也遵循了开闭原则。

实现方式

  • 使用产品类型和产品变体作为维度，创建一个矩阵。

  • 为所有产品定义一个抽象产品接口，并让所有具体产品类实现该接口。

  • 声明一个抽象工厂接口，在接口中为所有抽象产品提供一组构建方法。

  • 实现具体工厂类来创建每种产品变体的实例。

  • 在应用程序中编写初始化代码。根据应用程序配置或当前环境，初始化特定的具体工厂类，并将该工厂对象传递给需要创建产品的类。

  • 定位代码中所有直接调用产品构造函数的地方，并将其替换为调用工厂对象中相应的构建方法。

总结优缺点

优点：

  • 确保同一工厂生成的产品相互匹配，提供了产品的一致性。
  • 避免了客户端与具体产品代码的紧耦合，增强了代码的灵活性和可维护性。
  • 符合单一职责原则，将产品生成代码抽象到同一位置，使得代码更易于维护。
  • 遵循开闭原则，当引入新的产品变体时，无需修改客户端代码。

缺点：

  • 在Java中使用依赖注入时，可能会隐藏服务类的依赖关系，导致运行时错误，而这些错误在编译时无法捕获。
  • 添加新的产品对象可能会比较困难，特别是需要修改抽象工厂接口和所有具体工厂类。
  • 引入了许多新的接口和类，导致代码变得更加复杂。

拓展

我们可以针对不同的机器工厂设计工厂模式。在这个示例中，我们创建了一个名为FactoryMaker的类，负责返回不同的工厂实例。客户端可以使用FactoryMaker来创建所需的具体工厂，该工厂将生产不同类型的具体对象（如海陆空工厂）。为了参数化工厂类型，我们还使用了一个枚举。

这种设计可以提供更大的灵活性和可扩展性。客户端不需要直接与具体工厂类进行交互，而是通过FactoryMaker来获取所需的工厂对象。这样，客户端与工厂的耦合性较低，并且可以方便地切换和扩展不同类型的工厂。

通过引入枚举来参数化工厂类型，我们使得工厂类型的选择更加简单和一致。客户端只需要使用枚举值来选择所需的工厂类型，而无需处理复杂的条件逻辑。

这种工厂模式的设计可以增强代码的可读性、可维护性和可测试性。它符合开闭原则，当需要添加新的工厂类型时，不需要修改现有的客户端代码，只需要扩展FactoryMaker和相应的具体工厂类即可。
