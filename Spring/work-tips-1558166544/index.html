日常踩坑-Spring篇(Transaction rolled back because it has been marked as rollback-only)

异常信息：

事务的注解使用起来很简单，但是如果只了解皮毛就会出现事务失效、事务异常等问题（比如此时此刻的我）。出现上述问题的原因是：事务的嵌套过程中，嵌套的事务发生异常，抛出Exception时，被上层逻辑try-catch处理。

报错原因

在Spring框架中，如果一个方法A和方法B都被标记为REQUIRED(@Transactional默认REQUIRED)事务传播属性，当方法A调用方法B时，Spring会将它们的事务合并为一个。

在整个方法执行过程中，只有当每个子方法都没有出现异常时，整个方法执行完毕后才会将事务提交。

然而，如果某个子方法发生了异常，Spring会将该事务标记为rollback only，表示只能回滚事务而不能提交。但是，如果这个子方法没有将异常向上抛出，或者父方法捕获了子方法抛出的异常，那么该异常将不会触发事务回滚。事务将在整个方法执行完毕后提交，这可能导致Transaction rolled back because it has been marked as rollback-only异常的出现。这个异常表示事务被标记为只能回滚，但是在事务提交时却执行了提交操作，从而引发了异常。

解决方案

1.父方法不要捕获异常

2.子方法的事务propagation属性换为REQUIRES_NEW

Propagation的几种属性：

属性 功能描述
Propagation.REQUIRED 如果当前存在一个事务，那么该方法将在该事务中运行；否则，将创建一个新的事务。该方法的执行要么参与当前事务，要么创建一个新事务。
Propagation.SUPPORTS 如果当前存在一个事务，那么该方法将在该事务中运行；否则，它可以在非事务状态下运行。
Propagation.MANDATORY 该方法必须在一个已存在的事务中执行，否则将抛出异常。
Propagation.REQUIRES_NEW 无论当前是否存在事务，该方法都将创建一个新的事务。如果当前存在事务，则挂起该事务，直到新事务完成。
Propagation.NOT_SUPPORTED 该方法将在非事务状态下运行。如果当前存在事务，那么该事务将被挂起，直到该方法执行完毕。
Propagation.NEVER 该方法必须在非事务状态下执行。如果当前存在事务，则抛出异常。
Propagation.NESTED 表示如果当前已经存在一个事务，那么该方法将会在嵌套事务中运行。嵌套的事务可以独立于当前事务进行单独地提交或回滚。如果当前事务不存在，那么其行为和 Propagation.REQUIRED 效果一样。

以上是几种常用的事务传播属性（Propagation）及其功能描述。根据具体需求和业务逻辑，可以选择适合的事务传播属性来控制事务的行为。每种属性都有不同的含义和用途，可以根据需要灵活地使用它们。
